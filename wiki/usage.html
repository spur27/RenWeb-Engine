Runtime Requirements

RenWeb uses the platform's native WebView instead of bundling a browser, keeping executables under 2MB. You need to ensure the appropriate WebView runtime is installed on your system.

Linux - WebKitGTK 4.1

Required Runtime Libraries

    n
  • libwebkit2gtk-4.1-0 - WebKitGTK 4.1 WebView runtime (JS engine, DOM, networking)
  • libgtk-3-0 - GTK 3 windowing and widget toolkit
  • gstreamer1.0-plugins-base - Core GStreamer codecs (Ogg, Vorbis, Theora)
  • gstreamer1.0-plugins-good - Common formats (MP3, FLAC, WebM, Opus)

Recommended: WebGL & Hardware Acceleration

  • libegl1 - EGL interface for hardware-accelerated rendering
  • libgl1-mesa-dri - Mesa DRI drivers (OpenGL / WebGL)

Recommended: Media Playback

  • gstreamer1.0-plugins-bad - Extended formats (H.264, AAC, HLS)
  • gstreamer1.0-gl - GStreamer OpenGL video output
  • gstreamer1.0-libav - FFmpeg-backed codec support (widest format coverage)

Windows 7, 8, and 8.1

Download and install the WebView2 Runtime:

  1. Visit Microsoft WebView2 Downloads
  2. Download the Evergreen Standalone Installer
  3. Run the installer (requires administrator privileges)
  4. May require Universal C Runtime update

⚠️ Limited Support

Windows 7, 8, and 8.1 have reached end-of-life. While RenWeb will work, these systems no longer receive security updates. Windows 10 or later is recommended.

macOS - WKWebView

Built-in WKWebView

  • Native macOS WebView (included with system)
  • Minimum: macOS 10.15 Catalina

✅ No Installation Required

WKWebView is part of macOS 10.15 Catalina and later, and is automatically available. RenWeb applications work out of the box!

❌ Not Supported

RenWeb requires macOS 10.15 Catalina or later. Earlier versions (High Sierra, Mojave) do not meet the minimum requirements.

Using the RenWeb Engine from the Terminal

Should you want to run the application from the terminal to view logs, debug, or simply see what the engine has to say, you can run it from a terminal with various command-line arguments to test, adjust, and play around with the engine.

Command-Line Arguments

General Options

  • -h, --help - Display help information
  • -v, --version - Display version from info.json

Logging Options

  • -s, --log-silent - Suppress console log output
  • -b, --log-boring - Disable colored log output
  • -l, --log-level <n> - Set log level (0=trace, 5=critical, default=2)
  • -c, --log-clear - Clear log.txt before starting

Page Options

  • -p, --page <name> [names...] - Open specific page(s)

Usage Examples

# Open default pages from info.json
./renweb

# Open specific page
./renweb --page myapp

# Open multiple pages
./renweb -pwindow1 --page window2 -pwindow3

# Run with trace logging
./renweb --log-level 0

# Disable colored output
./renweb --log-boring

# Clear logs and run silently
./renweb --log-clear --log-silent

Note: The engine looks for info.json to retrieve the version number and title; that json file in particular is crucial for the engine to run.

File Structure

  • ● Executable files
  • ● Required files/directories
  • ● Auto-generated files
  • ● Optional directories
  • ● Bundle-specific (for dependencyless mode)
  • ● Comments
build/ ├── renweb # Executable ├── bundle_exec # Bundle environment script (.sh or .bat) - for bundled libs ├── info.json # Application metadata - REQUIRED ├── config.json # Window configuration - auto-generated ├── log.txt # Application logs - auto-generated ├── content/ # Page content - REQUIRED │ └── <page_name>/ │ ├── index.html # Required entry point │ └── ... ├── assets/ # Shared assets │ └── ... ├── resources/ # Resources │ └── ... ├── custom/ # User customization - optional │ └── (overrides for content/assets) ├── backup/ # Fallback content - optional │ └── (emergency fallback files) ├── plugins/ # Plugin libraries - optional │ └── (.so | .dll | .dylib) ├── lib/ # Bundled dependencies - for bundle mode │ └── (.so | .dll | .dylib) └── licenses/ # License files └── ...

RenWeb looks for files relative to the executable location. The file structure is modular, self contained, and designed for easy customization and interaction for both the programmer and end user.

File Resolution Order

When handling GET requests, the server searches in this order:

  1. custom/[page_name]/[file] - User customization (highest priority)
  2. content/[page_name]/[file] - Page-specific content
  3. [file] - Root directory (for assets, info.json, etc.)
  4. backup/[page_name]/[file] - Fallback content (lowest priority)

Example: Request for /assets/logo.png on page "test"

  1. Checks custom/test/assets/logo.png
  2. Checks content/test/assets/logo.png
  3. Checks assets/logo.png ← Found!
  4. Would check backup/test/assets/logo.png if not found
  5. Returns 404 if not found anywhere

This allows users to customize your application by placing files in custom/ without modifying the original content.

Required Fields

  • title (string) - Application name (displayed with -v flag)
  • version (string) - Application version
  • starting_pages (array or string) - Page name(s) to open on launch

Optional Fields

  • author (string) - Creator/developer name
  • description (string) - Brief application description
  • license (string) - License identifier (e.g., "BSL", "MIT", "GPL-3.0")
  • repository (string) - Source code repository URL
  • engine_repository (string) - Source code repository URL for engine (default is this)
  • plugin_repositories (array of strings) - Source code repository URLs for plugins
  • categories (array of strings) - Application categories for app stores/launchers (e.g., ["Utility"], ["Game", "Adventure"])
  • copyright (string) - Copyright notice
  • app_id (string) - Unique application identifier for packaging (reverse domain notation recommcended)
  • pkg_id (string) - Short package identifier for package management systems
  • startup_notify (boolean) - Whether to show desktop startup notification
  • port (number) - Port (default: random)

Permissions Object

  • geolocation (boolean) - Allow location access
  • notifications (boolean) - Allow desktop notifications
  • media_devices (boolean) - Allow camera/microphone
  • pointer_lock (boolean) - Allow mouse pointer locking
  • install_missing_media_plugins (boolean) - Auto-install codecs
  • device_info (boolean) - Allow hardware information access

Origins Array

The origins field contains an array of allowed base URLs with protocols. This provides URI allowlisting for security. 

"origins": [
  "https://example.one",
  "http://example.two/sequel",
  "*" // Allow all origins (not recommended)
]

Window Properties

All window properties are get/set-able via the client JavaScript API:

  • title (string) - Window title text (overrides title in info.json)
  • title_bar (boolean) - Show window title bar
  • size (object) - Window dimensions
    • width (number) - Width in pixels
    • height (number) - Height in pixels
  • position (object) - Window position on screen
    • x (number) - X coordinate in pixels
    • y (number) - Y coordinate in pixels
  • keepabove (boolean) - Keep window on top of others
  • resizable (boolean) - Allow window resizing
  • minimize (boolean) - Window minimized state
  • maximize (boolean) - Window maximized state
  • fullscreen (boolean) - Window fullscreen state
  • taskbar_show (boolean) - Show window in taskbar/dock
  • opacity (number) - Window opacity (0.0 = transparent, 1.0 = opaque)
  • initially_shown (boolean) - Show window immediately on load
  • merge_defaults (boolean) - Merge page settings with defaults on initial load

Default Settings

The __defaults__ section provides fallback values for all pages. Individual page settings override these defaults unless merge_defaults is set to false.

Tip: initially_shown

If set to false, the window won't show until you call Window.show() from JavaScript. This is useful to prevent the white flash during page load: 

window.onload = async () => {
  await Window.show();
};
