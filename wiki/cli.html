RenWeb CLI Tool (rw)

The rw command-line tool scaffolds, runs, builds, and packages RenWeb applications. It handles everything from project creation to cross-platform installer generation, without requiring you to touch makefiles or packaging scripts directly.

Installation

# Install globally via npm
npm install -g renweb-cli

# Verify
rw --help

Command Summary

CommandPurpose
rw createScaffold a new RenWeb project from a template
rw fetchDownload the engine executable, bundle, plugins, or JS API
rw runLaunch the engine from build/ (bundles, cached exe, or live download)
rw buildPrepare build/ for launch — copy pages, seed JS modules, fetch the engine; delegate to Vite build for framework projects
rw updateUpdate the engine, JS modules, npm/bun packages, and plugins in an existing project
rw addAdd a page or plugin: rw add page <name> | rw add plugin <url>
rw removeRemove a page or plugin: rw remove page <name> | rw remove plugin <name>
rw listList pages and/or plugins. Optionally filter: rw list pages | rw list plugins
rw packageGenerate installer packages for all platforms
rw dockerBuild, rebuild, or kill the renweb-cli Docker image
rw doctorCheck environment health and tool availability
rw docOpen documentation pages: rw doc js, rw doc usage, etc.

engine_repository — Custom engine source

All commands that download engine files (executables, bundles, API modules, plugin headers) read the engine_repository field from the project's info.json. When set, it overrides the default https://github.com/spur27/RenWeb-Engine source for all GitHub queries in that project.

{
  "engine_repository": "https://github.com/my-org/my-renweb-fork"
}

rw create

Scaffolds a brand-new RenWeb project. Prompts for project metadata (info.json fields), downloads the engine executable for the host platform, and sets up the project structure. The type argument is optional — omitting it defaults to vanilla.

Usage

rw create [type] [flags]

Project Types

TypeDescription
vanilla (default)Plain HTML/CSS/JS — no build step, no Node dependency
reactVite + React template
vueVite + Vue template
svelteVite + Svelte template
preactVite + Preact template
pluginScaffold a C++ plugin project
engineClone the full RenWeb Engine repository (with submodules by default)

Flags

FlagDescription
-y, --yesAccept all defaults without prompting
--bundleDownload a bundled release (includes shared libs) instead of the bare executable
--skip-submodulesSkip --recurse-submodules when cloning with engine type
--nodeAdd package.json with renweb as an npm dependency (vanilla only)
--denoAdd deno.json with jsr:@renweb/api as an import (vanilla only)

Examples

# Default vanilla project — no type required
rw create

# Vanilla project, accept all defaults
rw create --yes

# Vanilla project with npm renweb package (requires bundler for API imports)
rw create --node

# Vanilla project with Deno renweb package
rw create --deno

# React project in the current directory
rw create react

# Plugin project scaffold
rw create plugin my-plugin

# Clone the engine repository (includes submodules)
rw create engine

# Clone without recursing submodules
rw create engine --skip-submodules

What gets created (vanilla)

  • info.json / config.json — project metadata and window config
  • build/ — engine executable, content directory, and starter page
  • src/ — empty source directory for your application files
  • licenses/, resource/, credentials/ — fetched from the engine repository
  • .github/copilot-instructions.md — AI context for the RenWeb JS API and project structure
  • package.json (with --node) or deno.json (with --deno)

rw fetch

Downloads engine assets from the latest GitHub release into the current working directory. Without any flags, downloads the bare engine executable for the host platform.

Usage

rw fetch [flags]

Flags

FlagDescription
--executableDownload the bare engine executable (default when no flag is given)
--bundleDownload the bundled release archive (.tar.gz) and extract it
--pluginDownload plugin assets into the current directory
--apiDownload the JS API type definitions and source files (index.js, index.d.ts, index.ts)
--version <tag>Pin to a specific release tag instead of the latest (e.g. --version 0.0.6)

Flags can be combined. For example, rw fetch --api --executable downloads both the engine and the API files in one call.

Examples

# Download the engine executable (host platform auto-detected)
rw fetch

# Download a full bundled release
rw fetch --bundle

# Download JS API type definitions
rw fetch --api

# Download engine and API together
rw fetch --executable --api

# Pin to a specific release version
rw fetch --version 0.0.6

rw run

Handles the full development loop automatically. Project type is auto-detected; both vanilla and Vite projects are supported. If a bundle_exec launcher is present in build/ it is preferred over the bare executable; missing executables are resolved from the .rw/executables/ cache or downloaded from GitHub on first use. Vite projects also start watch mode and wait for the initial bundle. All processes tear down cleanly on Ctrl+C.

Usage

rw run

No flags. The host OS and architecture are detected automatically.

What it does

  1. Kills any engine instance whose PID is stored in build/.engine.pid
  2. Prefers bundle_exec.sh / bundle_exec.bat in build/ over the bare executable
  3. If no executable is present in build/, checks .rw/executables/ cache then downloads from GitHub
  4. For Vite projects: starts watch mode via the project's package manager and waits (up to 60 s) for the initial output bundle
  5. Launches the engine (or bundle launcher), saves its PID to build/.engine.pid
  6. On exit (Ctrl+C or engine crash): kills the Vite watcher and removes the PID file

Examples

# From anywhere inside the project
rw run

# Typical vanilla workflow
rw build && rw run

Vanilla projects

The Vite watch step is skipped entirely. rw run kills the previous engine instance and relaunches it immediately.

rw build

Prepares build/ for launch. For vanilla projects, selectively rebuilds the output directory — copying pages, seeding JS modules, and ensuring the engine executable is present. For Vite-based projects (React, Vue, Svelte, Preact), delegates to vite build via the project's package manager.

Usage

rw build [flags]

Flags

FlagDescription
--bundleDownload a bundle release (executable + lib/ + bundle_exec launcher) instead of the bare executable. Archive is cached in .rw/bundles/.

What vanilla build does

  1. Selectively clears build/content/, build/licenses/, build/resource/, build/assets/, build/info.json, and build/config.json. The engine executable, lib/, lib-*/, plugins/, and bundle_exec* are never removed.
  2. Copies info.json and config.json → build/
  3. Copies each page directory from src/content/ → build/content/, seeding src/modules/ into each page's modules/ subdirectory
  4. Copies src/assets/ → build/assets/, licenses/ → build/licenses/, resource/ → build/resource/
  5. Ensures the engine executable is present: checks .rw/executables/ cache first, downloads from GitHub and caches on first use
  6. With --bundle: fetches the bundle archive to .rw/bundles/ cache and extracts lib/, bundle_exec*, and the engine executable into build/

Examples

# Vanilla: copy pages + ensure engine
rw build

# Vanilla: use a bundled release (needed when system libs differ)
rw build --bundle

# Build then run
rw build && rw run

Active development

For a watch-mode workflow, use rw run directly — for Vite projects it starts Vite in watch mode and launches the engine together, with automatic process management.

rw update

Updates the project to the latest engine release. Also runs the package manager install step (npm/bun/yarn/pnpm), re-downloads vanilla JS API modules from GitHub, and updates all plugins listed in info.json["plugins"]. Automatically detects whether the project uses a bare executable or a bundle. On macOS, always behaves as --executable-only for the engine.

What it does

  1. Runs the project's package manager install (npm install, bun install, etc.) if a package.json or deno.json is present
  2. Re-downloads src/modules/renweb/index.js and index.d.ts from the engine repository (vanilla projects only)
  3. Downloads the engine executable (bare or bundle) from the specified or latest release, replacing the old version in build/
  4. Updates plugins from their GitHub release assets: downloads to .rw/plugins/ cache and copies the host-arch binary to build/plugins/
  5. Syncs the version field in info.json to match the downloaded release

Usage

rw update [flags]

Flags

FlagDescription
--bundle-onlyUpdate the bundle archive only (skip the bare executable)
--executable-onlyUpdate the bare executable only (skip bundle artifacts)
--version <tag>Update to a specific release tag instead of the latest (e.g. --version 0.0.6)

Examples

# Auto-detect and update everything
rw update

# Update only the executable, keep current bundle libs
rw update --executable-only

# Pin to a specific version
rw update --version 0.0.6

rw add

Adds a page or plugin to the project. Pages are scaffolded in src/content/ (vanilla) or build/content/ (Vite) and registered in config.json. Plugins are cloned, built, and installed to build/plugins/.

Usage

rw add <page|plugin> [args]

Types

TypeArgsDescription
page<name> [--starting-page]Scaffold index.html in the content directory and register in config.json. Use --starting-page to also add the page to info.json["starting_pages"]
plugin<repo-url>Clone the plugin repo, run make or cmake, copy the resulting shared library into build/plugins/

Examples

# Add a new page named "settings"
rw add page settings

# Install a plugin from GitHub
rw add plugin https://github.com/example/my-renweb-plugin

Notes

  • Page names are normalised to snake_case (spaces and hyphens become underscores)
  • The generated index.html includes a <script type="module" src="./modules/renweb/index.js"> tag
  • For plugin: git must be available, and the repo must have a Makefile or CMakeLists.txt
  • The built library must export CreatePlugin(std::shared_ptr<ILogger>)

rw remove

Removes a page or plugin from the project. The page's source directory is moved to .rw/trash/ (never permanently deleted); the build/content/ copy is removed along with the config.json entry and any reference in info.json["starting_pages"]. Plugin shared libraries are removed from build/plugins/ (cached sources in .rw/plugins/ are kept).

Usage

rw remove <page|plugin> [args]

Types

TypeArgsDescription
page<name>Move src/content/<name> to .rw/trash/, remove build/content/<name>, and clean the config.json entry. Refuses to remove __defaults__
plugin<name>Remove a plugin library from build/plugins/ by name (fuzzy match on filename/stem)

Examples

# Remove the "settings" page
rw remove page settings

# Remove a plugin by name (without extension)
rw remove plugin my-renweb-plugin

rw list

Lists pages and/or installed plugins. Without an argument, shows both. Pass pages or plugins to filter the output.

Usage

rw list [pages|plugins]

Output

FilterDescription
(none)Show both pages and plugins
pagesPages found in the content source directory — ★ = starting page (from info.json["starting_pages"])
pluginsPlugins from info.json["plugins"] — [✓] = in build/plugins/, [○] = cached only, [⚠] = not found

Examples

# Show all pages and plugins
rw list

# Show only registered pages
rw list pages

# Show only installed plugins
rw list plugins

rw package

Generates installer and distribution packages for Windows, macOS, and Linux from a staged RenWeb build. Supports every major packaging format on each platform.

Usage

rw package [flags]

Flags

FlagDefaultDescription
--os <os>host OSTarget OS: windows, macos, linux
--arch <arch>host archTarget architecture: x86_64, arm64, x86_32, …
--ext <ext>allLimit to a single package format (e.g. deb, nsis, flatpak)
--cache <dir>temp dirDirectory used to cache downloaded engine assets
--bundle-only—Stage a bundled engine release (includes shared libs)
--executable-only—Stage the bare executable only
--engine-repo <url>GitHub releasesOverride the source repository URL for engine downloads
--plugin-repo <url>—Additional plugin repository to bundle with the package
--gpg-sign <key>—GPG key ID to sign Linux packages (deb, rpm, Flatpak)
--sign-identity <id>—macOS Developer identity for code-signing (codesign)
--sign-cert <path>—Windows code-signing certificate (.pfx) path

Supported Formats by Platform

PlatformFormatTool required
WindowsNSIS installer (.exe)makensis
MSI (.msi)wix
MSIX (.msix)makeappx
Chocolatey / NuGet / winget manifests—
macOSDMG disk image (.dmg)create-dmg
PKG installer (.pkg)pkgbuild, productbuild
LinuxDebian package (.deb)fpm
RPM package (.rpm)fpm
Arch package (.pkg.tar.zst)fpm
AppImage (.AppImage)appimagetool
Flatpak (.flatpak)flatpak-builder

Examples

# Package for the host platform (all formats)
rw package

# Build only a .deb for the current machine
rw package --os linux --ext deb

# Build a Windows NSIS installer (cross-platform where makensis is available)
rw package --os windows --ext nsis

# Package for Linux arm64 with a bundled engine
rw package --os linux --arch arm64 --bundle-only

rw docker

Manages the renweb-cli Docker image used by the packaging pipeline. Use this to build the image for the first time, rebuild it after cli/Dockerfile changes, or kill stuck containers.

Usage

rw docker <action>

Actions

ActionDescription
buildBuild the renweb-cli image from cli/Dockerfile
rebuildRebuild the image with --no-cache (forces a full layer rebuild)
killKill all running containers based on the renweb-cli image

Environment

Set RENWEB_IMAGE to override the default image name (renweb-cli):

RENWEB_IMAGE=my-image rw docker build

Examples

# Build the Docker image for the first time
rw docker build

# Rebuild after changing Dockerfile (clears layer cache)
rw docker rebuild

# Kill all running renweb-cli containers
rw docker kill

rw doctor

Inspects the local environment and the current RenWeb project for common problems. Reports which required and optional tools are present, checks Node.js version compatibility, and verifies the project structure.

Usage

rw doctor

No flags. Exit code is 0 if all required checks pass; 1 if any required item is missing or misconfigured.

rw doc

Opens RenWeb documentation pages in the system default browser. Without arguments, opens the home page. Pass one or more page names to jump directly to specific pages — multiple names open multiple tabs.

Usage

rw doc [page...]

Pages

NameOpens
(none)Home — ?page=home
homeHome — ?page=home

usageUsage guide (run / build / dev workflow) — ?page=usage
cliCLI command reference — ?page=cli
compilationBuilding the engine from source — ?page=compilation
download, downloadsDownloads — ?page=downloads
plugin, pluginsPlugin development reference — ?page=plugins

Examples

# Open docs home
rw doc

# Open JS API reference
rw doc js

# Open multiple pages at once
rw doc js usage cli
